#### 리스트에서 중복된 숫자 찾기
---

정수가 담긴 리스트가 있다. 

1. 정수는 1에서 n까지 수 중 하나이다. 
2. 리스트의 길이는 n+1이다. 

따라서 리스트에는 적어도 하나의 정수가 적어도 두 번 존재하게 된다. 하지만 숫자는 여러 번 중복되어 존재할 수 있고, 두 번 이상 중복되어 나타나게 된다. 

#### Gotchas
---

이것을 O(1) 공간으로 해결할 수 있다. 

O(1) Space를 유지하면서 O(n^2) 시간보다 적은 시간으로 해결할 수 있다. 

O(1) Space 와 O(nlogn) time으로 해결할 수 있다. 

대부분의 O(nlogn) 알고리즘은 무엇인가를 두 배로 늘리거나 절반으로 자른다. 이 때 우리는 어떻게 리스트를 한 번씩 이터레이션 돌 때마다 숫자들의 절반을 걷어낼 수 있을까?

#### Breakdown
---

알겠지만, 이건 고전문제이다. 

단순히 리스트를 탐색하면서 set을 이용해 어떤 숫자가 이미 존재한 숫자였는지를 확인하면 된다. 

```python
def find_repeat(numbers):
    numbers_seen = set()
    for number in numbers:
        if number in numbers_seen:
            return number
        else:
            numbers_seen.add(number)
    
    raise Exception('no duplicate!')
```

완성! 했지만 O(n) time, O(n) space 이다. 

Space를 최적화 시켜줬어야 한다. 

O(n)은 Space 관점에서 사실 꽤 크다. 그렇다면 어떻게 O(1) Space로 풀 수 있을까?

"Brute Force" 방식으로 풀 수 있을 텐데, 1부터 n까지의 숫자 하나씩에 대해 리스트를 탐색하면서 그것이 두 번 등장하는지 확인하는 것이다. 

```python
def find_repeat_brute_force(numbers):
    for needle in range(1, len(numbers)):
        has_been_seen = False
        for number in numbers:
            if number == needle:
                if has_been_seen:
                    return number
                else:
                    has_been_seen = True

    raise Exception('no duplicate!')
```

이건 O(1) Space와 O(n^2) Time 이다. 

이 이상 Space를 줄일 수는 없다. 그런데, 시간이 꽤 오래 걸린다. 이를 더 개선할 수 있을까?

O(n^2) time을 줄이는 한 가지 방법은 O(nlogn) time으로 푸는 것이다. 

정렬이 O(nlogn) time이 소요된다. 그리고, 만약 리스트를 정렬한다면, 중복된 수들은 서로 나란히 있게 될 것이다!

하지만 리스트를 정렬하려고 해보면 정렬된 리스트를 저장하기 위해 O(n) Space가 필요하게 될 것이다. 

만약 인풋 리스트를 ***in place***로 정렬하지 않는다면 말이다!

자 그럼, 다음의 작업을 해주면 될텐데:

1. 리스트를 in-place 정렬해준다.(예를 들자면 in-place merge sort)

2. 정렬된 리스트를 왼쪽에서 오른쪽으로 탐색한다.

3. 숫자가 동일한 인전한 두 수를 찾는 즉시 리턴한다.

이렇게 한다면 O(1) Space와 O(nlogn) Time으로 풀 수 있을 것이다. 

하지만 인풋을 변경해버리는 것은 바람직하지 않다. - 그건 코드 내의 어디선가 문제를 일으킬 수도 있다. 

인풋을 변경하지 않고도 이 시간과 공간복잡도를 유지할 수 있을까?

잠깐 물러서서 문제를 바라보자. **어떻게하면 이 문제를 더 작은 문제로 쪼갤 수 있을까?**

O(nlogn) 시간에 해결하고자 한다면, 아마도 반복적으로 무언가를 두 배로 늘리거나 반복적으로 무언가를 반으로 자르게 될 것이다. 

그것이 보통 "logn"을 얻는 방식이다. 그럼 만약 문제를 어떻게 해서든 절반으로 자를 수 있다면 어떨까?

글쎼, 리스트를 절반으로 자른 뒤 어떤 쪽에 정답이 있을지 알아냄으로써 문제를 자른다면, 이분탐색이 적당할 수도 있다. 

하지만 이분탐색에서 둘 중 어느 절반에 정답이 있다고 자신있게 이야기할 수 있는 이유는 리스트가 정렬되었기 때문이다. 

이 문제에서, 인풋으로 들어온 정렬되지 않은 리스트를 반으로 자른다고 해서 어떤 원소가 왼쪽 절반 혹은 오른쪽 절반에 존재한다고 확실하게 이야기할 수가 없다. 

만약 리스트를 절반으로 자르는 것 말고, 다른 방법으로 이 문제를 절반으로 자를 수 있다면 어떨까?

이 문제에서, 우리는 모래사장(인풋 리스트)에서 바늘(반복되는 숫자)를 찾고 있는 것이다. 

만약 모래사장을 절반으로 자르는 대신에, ***바늘에 대한 가능성들의 집합***을 절반으로 자르면 어떨까?

우리가 찾는 숫자에 대한 가능성의 모든 범위는 1부터 n까지이다. 실제 그 숫자가 1부터 n/2까지의 범위 혹은 n/2+1 부터 n까지의 범위 중 어디에 있는지 어떻게 검증할 수 있을까?

우리가 범위를 어떻게 정의하고 있는지 빠르게 점검해보자. 우리가 n/2를 했을 때, 우리는 정수 나눗셈을 하고 있기 때문에 나머지는 버리게 된다. 

어떻게 되는지 보려면, n이 짝수일 때와 홀수일 때를 살펴보면 된다

- 만약 n이 6이면(n이 짝수), ```n/2 = 3```이고 ```n/2 + 1 = 4```이므로, 따라서 범위는 (1~3), (4~6)이 된다. 

- 만약 n이 5이면(n이 홀수), ```n/2 = 2```(나머지는 버린다) 이고 ```n/2 + 1 = 3```이므로, 따라서 범위는 (1~2), (3~5)가 된다. 

따라서 나눠지게 되는 범위에 대한 몇가지 특징을 발견할 수 있다:

1. 범위들은 반드시 동일한 크기는 아니다. 

2. 범위들은 겹쳐지지 않는다. 

3. 두 범위를 합치면, 원래 인풋 리스트의 범위가 1부터 n까지임을 의미한다. 즉 수학적으로, 그들의 합집합이 1~n이라고 할 수 있다. 

그럼, 우리가 찾는 숫자가 1 ~ n/2 에 있는지 n/2+1 ~ n 에 있는지 어떻게 알 수 있을까?

원래 문제가 명시했던 것을 생각해보자. 우리는 리스트가 n+1 개의 원소를 가지고 있기 때문에 적어도 하나가 반복된다는 것을 알 수 있었고, 그 숫자들은 모두 n개의 고유한 정수를 가지는 1부터 n 사이에 있다. 

"우리가 가질 수 있는 것보다 더 많은 원소를 가지고 있으니, 적어도 한 번의 중복이 발생할 것이다" 라는 것은 아주 강력하다. 

이는 때때로 "비둘기집 원리"라고도 불린다. 

> **비둘기집 원리**란 m개의 박스에 n개의 물건이 있고, n > m이라면, 적어도 한 개의 박스는 한 개 이상의 물건을 담고 있다는 원리이다. 예를 들면, 세 글러브 그룹에는 적어도 두 개의 왼쪽 글러브 혹은 두 개의 오른쪽 글러브가 있어야 한다는 것이다. 

우리가 가진 범위의 어떤 절반이 반복된 값을 포함하는지 보기 위해 비둘기집 원리를 이용할 수 있을까?

인풋 리스트를 두 개의 서브 리스트로 나눴다고 가정해보자. 한 개는 1부터 n/2까지를, 나머지 하나는 n/2+1부터 n까지를 담고 있다. 

각 서브 리스트는 원소의 개수 뿐만 아니라 가능한 유일한 정수의 개수 또한 가지고 있다.(즉, 그것이 가지는 가능한 정수의 범위의 길이)

원래 인풋 리스트에서 원소의 개수 vs 가능한 유일한 정수의 개수에 대해 우리가 알고 있는 것을 감안할 떄, 이 서브 리스트들에서의 원소의 개수 vs 가능한 유일한 정수의 개수에 대해서는 어떻게 말할 수 있을까?

서브 리스트들의 원소 개수의 합은 ```n+1```(인풋 리스트의 원소 개수)이고, 서브 리스트들의 가능한 유일한 정수 개수의 합은 ```n```(인풋 리스트의 가능한 유일한 정수 개수)이다. 

서브 리스트들의 원소 개수의 합이 서브 리스트들의 가능한 유일한 정수 개수의 합보다 반드시 1 커야 하기 때문에, 서브 리스트 중 하나는 그것이 가질 수 있는 고유한 정수보다 적어도 하나 더 많은 원소를 가지고 있어야 한다. 

납득이 되지 않는가? 우리는 이것을 모순으로 증명해낼 수 있다. 둘 중 어떤 리스트도 가능한 유일한 정수보다 더 많은 원소를 가지고 있지 않다고 가정해보자. 

다시 말해서, 두 리스트 모두 기껏해야 그들이 가진 고유한, 가능한 정수들 만큼의 원소 개수를 가진다. 그들의 원소 개수의 합은 따라서 최대 각각의 가능한 정수의 개수의 총합일 것이기 때문에, 결국 n이다. 

이는 모순이다. 원래의 인풋 리스트로부터 원소의 총 개수가 n보다 큰, n+1임을 알고 있기 때문이다. 

이제 우리는 서브 리스트들 중 하나가 가질 수 있는 고유한 정수보다 1개 또는 그보다 더 많은 원소를 가지고 있다는 것을 알기 때문에, 원래의 인풋 리스트가 적어도 한 개의 중복을 가지고 있다는 것을 알 때 사용했던 같은 비둘기집 원리로부터 서브리스트가 적어도 한개의 중복을 가지고 있어야 한다는 것을 알게 된다. 

따라서 어떤 서브 리스트가 그것의 가능한 고유한 정수의 개수보다 큰 원소의 개수를 가지고 있는지 안다면, 이 접근방법을 서브리스트들을 반으로 계속 잘라가면서 범위에 1개의 원소만 남을때까지 재귀적으로 수행할 수 있을 것이다.  

물론, 실제로 리스트를 서브 리스트들로 나눌 필요는 없다. 우리가 신경쓰고 있는 것은 각 서브 리스트의 길이가 어떻게 될 것인가에 대한 것이다. 

따라서 간단하게 인풋 리스트에 대한 탐색을 한 번 하면서, 각 서브리스트에 존재하게 될 원소의 개수를 카운트해주면 된다. 

이를 코드로 쿠현할 수 있겠는가?

**주의사항 - 재귀로 진행할 때, 콜스택에 공간 비용을 발생시킬 것이다.** 그 대신, 이터레이티브하게 해결해보자.


#### Solution
---

이 접근방식은 리스트를 절반으로 나누는 것이 아니라 각 단계에서 가능한 정답의 범위를 절반으로 나눈다는 것만 제외하면 이분탐색과 유사하다.

1. 인풋 리스트에서 1부터 n/2 사이 범위 내에 존재하는 정수의 개수를 찾는다. 

2. 그 값을 같은 범위에서 가능한 고유한 정수의 개수와 비교한다. 

3. 만약 실제 정소의 개수가 가능한 정수의 개수보다 크다면, 1부터 n/2범위에 중복이 존재한다는 것을 알 수 있으므로 그 범위에 대해 반복적으로 같은 작업을 수행한다. 

4. 만약 실제 정수의 개수가 가능한 정수의 개수보다 크지 않다면, n/2+1부터 n 범위에 중복이 존재한다는 것을 알 수 있으므로 그 범위에 대해 반복적으로 같은 작업을 수행한다. 

5. 어느 지점에서, 범위가 오직 1개의 정수값만을 포함할 것이고, 그것이 정답이 될 것이다. 


``` python
def find_repeat(numbers):
    floor = 1
    ceiling = len(numbers) - 1

    while floor < ceiling:
        midpoint = floor + ((ceiling - floor) / 2)
        lower_range_floor, lower_range_ceiling = floor, midpoint
        upper_range_floor, upper_range_ceiling = midpoint+1, ceiling

        # Count number of items in lower range
        items_in_lower_range = 0
        for item in numbers:
            # Is it in the lower range?
            if item >= lower_range_floor and item <= lower_range_ceiling:
                items_in_lower_range += 1
        
        distinct_possible_integers_in_lower_range = (
            lower_range_ceiling - lower_range_floor + 1
        )

        if items_in_lower_range > distinct_possible_integers_in_lower_range:
            # There must be a duplicate in the lower range
            # So use the same approach iteratively on that range
            floor, ceiling = lower_range_floor, lower_range_ceiling
        else:
            # There must be a duplicate in the upper range
            # So use the same approach iteratively on that range
            floor, ceiling = upper_range_floor, upper_range_ceiling

    # Floor and ceiling have converged
    # We found a number that repeats
    return floor

```

#### Complexity
---

O(nlogn) 시간과 O(1) 공간 

이 해결방법은 까다롭지만, 실제로 O(n) 시간으로 런타임을 줄이면서 O(1) space를 유지하도록 개선할 수 있다. 

이 방법은 NUTS;인데, 이건 대부분의 면접관이 기대하는 것의 범주를 넘어선 것일 수 있다.

#### 결론
---

우리의 풀이는 이분탐색을 수정한 것이다. 원하는 런타임에 대해 추론함으로써 풀이에 도달했다. 

1. "Brute Force" 풀이법의 O(n^2)으로 시작해 더 개선할 수 없을지 고민하였다. 

2. O(n^2)을 줄이기 위해서는 O(n)이나 O(nlogn)으로 해결해야 하기 때문에, O(nlogn) 런타임으로 해결할 수 있는 방법을 생각하였다. 

3. log(n)은 보통 반복적으로 절반을 잘라나감으로써 얻어지기 때문에, 그 아이디어를 실험해봄으로써 최종 알고리즘에 도달하였다. 

원하는 런타임에서 시작해 역으로 찾아나가는 방식은 모든 종류의 코딩 인터뷰 문제를 불 때 강력한 전략이 될 수 있다.
