### Garbage Collector(가비지 콜렉터)
------

**가비지 콜렉터**는 프로그램이 더 이상 사용하지 않는 메모리를 자동으로 비워준다. 

예를 들어, python 2.7버전에서 다음과 같이 실행하면:

```python
def get_min(nums):
    # Note: 이것은 최소값을 찾는 가장 빠른 방법은 *아니다*
    nums_sorted = sorted(nums)
    return nums_sorted[0]

my_nums = [5, 3, 1, 4, 6]
print(get_min(my_nums))
```

```get_mins()```내의 ```nums_sorted```를 살펴보자. 우리는 함수 내에서 전체 리스트를 할당했기 때문에, 함수가 리턴하고나면 이제 더이상 그 리스트가 필요하지 않다.  실제로, 함수가 리턴하고나면 우린 *그 리스트에 대한 참조를 더 이상 가지고 있지 않다.* 

메모리단에서 그 리스트에게 무슨 일이 생기고 있을까? Pyhon 2.7 버전 가비지 컬렉터는 우리가 더 이상 그것이 필요하지 않다는 것을 알아차리고 그 공간을 비운다. 

> 가비지 컬렉터는 무언가 비워질 수 있다는 것을 어떻게 알까? 
>
> 하나의 방법은 비워질 수 *없는* 것을 찾는 것이다. 예를 들면, 로컬변수는 나중에 필요할 수도 있기 때문에 확실히 비워질 수 없다. 그리고, 만약 리스트를 가지고 있다면, 그 리스트의 엘리먼트들은 어느 것 하나라도 비워져서는 안 될 것이다.
>
> 가비지 컬렉터 전략에 숨어있는 핵심은 다음과 같다:
>
> 1. 메모리에 있는 것 중 어떤 것이 쓰이고 있는지, 혹은 나중에 쓰일 수 있는지 신중하게 파악한다. 
> 2. 그 외 모든 것을 비운다. 
>
> 이 전략을 **tracing garbage collection**이라 하는데, 이 전략의 첫 번째 단계를 하나의 object(예를 들어 리스트)에서 그 다음(리스트 내의 엘리먼트)에 대한 참조를 추적하면서 실행되기 때문이다. 
>
> 또다른 방법은 각 object를 참조하는 것들의 개수를 추적하는 것이다. (예를 들면 Array의 공간을 차지하고 있는 변수나 그래프에서 같은 노드를 가리키고 있는 여러 엣지들)  이 숫자를 object의 참조 횟수(**reference count**)라고 한다. 
>
> 이 경우에, 가비지 컬렉터는 참조 횟수가 0인 것을 모두 비우게 된다. 
>
> 이 전략을 **reference counting**이라 하고,  각 object가 *참조*되는 횟수를 *카운트*하기 때문이다. 

C같은 몇몇 언어는, 가비지 컬렉터를 가지고 있지 않는다. 따라서 우리가 할당했다가 사용이 끝난 메모리를 수동으로 비워줘야 한다. 

```c
// terminating null byte('\0')을 포함해 
// 15글자를 가질 수 있는 String을 만든다.
str = malloc(15);

// ...코드...

// 사용이 끝났다. 메모리를 비워보자!
free(str);
```

우리는 때때로 이 방법을 **manual memory management(수동 메모리 관리)**라 부른다. C++같은 언어는, 수동 메모리 관리 방법과 자동 메모리 관리 방법을 둘 다 가지고 있다.
