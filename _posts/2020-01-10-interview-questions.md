# Interview

1. OOP

oop란 프로그램을 객체라는 기본 단위로 나누고, 이 객체들의 상호작용으로 서술하는 방식을 말한다. 장점으로는 클래스를 가져와 이용할 수 있고 상속을 통해 확장가능하기 때문에 코드 재사용성을 높일 수 있고, 수정할 때 클래스 내부에 멤버 변수 혹은 메서드만 수정하면되므로 유지보수가 쉽다는 점, 클래스 단위로 모듈화시켜서 개발할 수 있기 때문에 업무 분담이 쉬워 대형 프로젝트에 적합하다는 점이 있다. 

하지만, 처리속도가 상대적으로 느리고 객체가 많을 때 용량이 커지는 문제, 설계 시 많은 시간과 노력이 필요하다는 단점이 있다. 

클래스와 인스턴스, 캡슐화, 상속, 추상화, 다형성의 요소를 가지고 있다. 

클래스란 데이터를 만들기 위해 추상화를 거쳐 변수와 메서드로 정의한 것을 의미하고, 인스턴스(객체)란 클래스에서 정의한 것을 토대로 실제 메모리에 할당된 것으로 실제 프로그램에서 사용되는 데이터를 의미한다. 이 때, 추상화란 불필요한 정보는 숨기고 중요한 정보만을 표현함으로써 공통의 속성이나 기능을 묶어 이름 붙이는 것을 의미한다. 

캡슐화란 코드를 재수정없이 재활용하기 위해 관련된 기능과 특성을 한 곳에 모아 클래스를 만드는 것을 의미한다. 

상속은 부모클래스의 속성과 기능을 이어받아 사용하게 하고, 기능의 일부분을 변경해야 할 경우 상속받은 자식클래스에서 해당 기능만 오버라이딩하여 사용할 수 있게 하는 것이다. 

다형성은 하나의 변수명, 함수명 등이 상황에 따라 다른 의미로 해석될 수 있는 것이다. 

2. var, let, const 차이점

ES6로 넘어오면서 기존 ES5까지 사용하던 var에 let과 const가 추가되었다. 

var는 fuction 스코프를, const와 let은 block 스코프를 가진다. const와 let은 변수냐 상수냐의 차이이다. let은 값을 바꿀 수 있지만 const는 바꿀 수 없다. 여기서 블록 스코프란 function, if, for, while, switch 등 자바스크립트의 모든 스코프를 의미한다. 변수 선언 이전에도 변수를 사용할 수 있는 것을 호이스팅이라고 하는데, let은 호이스팅이 일어나지 않고, var은 호이스팅이 일어난다. 따라서 var로 선언한 함수는 코드 순서에 상관없이 함수 선언이 가능하다. IE10이하는 let을 지원하지 않고, IE11에서 let을 지원하기는 하지만, for문에서 루프를 돌때마다 변수가 만들어지지 않는 문제가 있다. 따라서 IE에서 let을 제대로 사용하기 위해서는 let을 var로 바꿔주는 변환기가 필요한데, 예를 들면 babel이 있다. 

3. 호이스팅이란

호이스팅이란, 모든 선언을 가장 위로 끌어올려 해당 함수 유효범위 최상단에 선언하는 것을 의미한다 즉, 변수 선언 이전에도 변수를 사용할 수 있는 것을 의미한다. 이 때, 변수나 함수의 선언만 위로 끌어 올려지고, 할당은 끌어 올려지지 않는다. 호이스팅은 함수선언문과 함수표현식에서 서로 다르게 작동하는데, 변수에 할당된 함수표현식은 끌어 올려지지 않기 때문에 이때는 변수의 스코프 규칙을 그대로 따른다. 함수표현식의 선언이 호출보다 아래에 있는 경우 에러가 발생하게 된다. 같은 이름의 var변수와 함수가 선언되었을 때, 변수 선언이 함수 선언보다 위로 끌어 올려진다. 그러나 값이 할당되어 있지 않은 변수의 경우, 함수선언문이 변수를 덮어쓴다. 

코드의 가독성과 유지보수를 위해 호이스팅이 일어나지 않도록 하는게 좋으며, 따라서 let과 const를 사용하는게 좋다. 

4. 비동기 프로그래밍이란

자바스크립트는 위에서 아래의 단반향으로 함수가 해석되며 실행되는 단일 스레드 언어이기 때문에, 필요한 데이터가 클라이언트 쪽으로 전달되지 않은 채로 사용자에게 보여지면 서비스상 큰 오류가 생길 수 있다. 이러한 문제점을 해결하기 위해 도입된 것이 비동기 프로그래밍이다. 비동기란 자바스크립트에서 정의한 코드가 완료되지 않아도 다음 코드를 실행하는 것을 말한다. 이 때, 코드의 실행 순서를 보장받기 위해서는 콜백을 사용한다. 

5. Ajax란? 

자바스크립트의 라이브러리 중 하나로, 브라우저가 가지고 있는 XMLHttpRequest 객체를 이용해서 전체 페이지를 새로 고치지 않고 페이지의 일부만을 위한 데이터를 로드하는 기법을 말한다. 즉, 클라이언트와 서버 간에 xml 데이터를 비동기 방식으로 주고받는 기술이다. 페이지 전체를 로드할 경우 자원과 시간의 낭비가 발생하는데, ajax는 일부분만 갱신할 수 있도록 XMLHttpRequest객체를 통해 서버에 request하는 것이다. 이 경우 Json이나 Xml 형태로 필요한 데이터만 받아 갱신하기 때문에 그만큼 자원과 시간을 아낄 수 있다. 따라서 웹페이지의 속도가 향상되고 서버의 처리가 완료될 때까지 기다리지 않고 처리가 가능하다. 하지만, 연속으로 데이터를 요청하면 서버 부하가 증가할 수 있고, 사용자에게 아무런 진행 정보가 주어지지 않기 때문에 요청이 완료되지 않았는데 사용자가 페이지를 떠나거나 오작동할 수 있다는 단점이 있다. 

6. Callback이란?

어떤 일을 다른 객체에게 시키고, 그 일이 끝나는 것을 기다리는 것이 아니라 그 객체가 나를 다시 부를 때까지 내 할 일을 하고 있는 것을 의미한다.즉, 어떤 작업을 요청한 후 다른 작업을 수행하다가 이벤트가 발생하면 그에 대한 응답을 받아 처리하는 것이다. 식당에서 대기자 명단에 이름을 올려놓고 돌아다니다가 자리가 나면 식당에서 나에게 연락이 오게 되는 것과 동일하다.  사용자가 호출 시점을 결정하는 것이 아니라, 시스템 입장에서 필요한 타이밍에 호출이 필요하다는 상황을 가정해보면, 특정 이벤트가 발생하기 전에는 프로그램 입장에서 이벤트의 정보를 사용자에게 알려줄 수 없고, 개발자가 개발 시에 사용자에게 알려줄 이벤트가 언제 발생할지도 모르게 된다. 이런 경우, 이벤트 발생시 데이터를 전달하게끔 콜백을 이용하면 된다. 

하지만, 여러 콜백함수를 중첩하여 사용하면 콜백지옥에 빠지게 되는데, 이는 promise나 async로 해결이 가능하다. 

7. promise란?

프로미스는 주로 서버에서 받아온 데이터를 화면에 표시할 때 사용한다. 만약 데이터를 받아오기도 전에 데이터를 다 받아온 것 처럼 화면에 데이터를 표시하려고 하면 오류가 발생하고나 빈 화면이 뜨게 되는데, 이를 해결하기 위한 방법 중 하나가 프로미스이다. 프로미스는 pending, fulfilled, rejected의 3가지 상태를 갖는다. new promise()메서드를 호출하면 pending상태가 되고, 콜백함수의 인자로 resolve, reject에 접근할 수 있다. resolve를 실행하면 fulfilled상태가 된다. 이 때, then()을 이용해 처리 결과 값을 받을 수 있다. 실패 상태가 되면 실패 처리의 결과 값을 catch로 받을 수 있다. 그리고, 페이지에 입력된 사용자 정보를 받아와 파싱, 인증 등의 작업을 거칠 때 여러 개의 프로미스를 .then()으로 연결해 처리할 수 있다. 

8. async & await이란?

비동기 처리 코드에서 기존의 비동기 처리 방식인 콜백 함수와 프로미스의 단점을 보완하고 가독성이 좋은 코드를 작성할 수 있게 도와주는 것이다. 여러개의 비동기 처리 코드를 다룰 때, 기존의 비동기 처리코드 방식으로 사고하지 않고 한 줄 씩 읽으면서 사고할 수 있으므로 가독성이 좋다. 예외 처리는 try catch 문법을 적용할 수 있다. 

9. 클로저란?

외부함수의 변수에 접근할 수 있는 내부 함수를 말한다. 외부함수가 리턴된 이후에도 여전히 내부함수가 외부함수의 변수에 접근할 수 있다. 보통 함수 내에서 사용된 로컬 변수는, 해당 함수의 실행이 종료되면 파기되지만, 클로저함수에 의해서 참조되고 있는 경우에는, 해당 로컬 변수를 파기하지 않고 계속 보관한다. 비동기 작업에 사용된다. 

10. CORS란?

Cross Origin Resource Sharing의 줄임말로, 크로스 도메인 요청을 가능하게 하는 것이다. 자바스크립트 보안상의 이유로 자신과 동일한 도메인으로만 http요청 보내도록 제한하였는데, 여러 도메인에 걸쳐 구성되는 대규모 웹 프로젝트가 늘어가고, REST API등을 이용한 외부 호출이 많아지자 이를 가능하게 한 것이 CORS다. 

11. repaint와 reflow

css 애니메이션 구현비용과 직접적으로 연관되어 있는 것으로, reflow와 repaint를 최소화해야 부드러운 애니메이션을 적용할 수 있다. reflow란 생성된 dom 노드의 레이아웃 변경 시 영향받는 모든 노드의 수치를 다시 계산해 렌더 트리를 재생성하는 작업이다. repaint는 reflow 과정이 끝난 후 재생성된 렌더 트리를 다시 그리는 작업으로 수치와 상관없는 background-color 등의 스타일 변경 시 reflow과정이 생략된 repaint작업만 수행한다. reflow 최소화하려면, 클래스 변화에 따른 스타일 변화를 원할 경우, dom 구조상 끝단에 위치한 노드에 클래스를 추가하거나, will-change를 검토해 볼 수 있다. 

 12. 브라우저는 어떻게 동작하는가?

사용자가 브라우저에 url(www.naver.com)을 입력하면, 

브라우저는 도메인이 캐시에 들어있는지 확인한다. 만약, 브라우저 캐시에서 못 찾으면, os에 저장된 dns 캐시를 찾게 된다. 만약, os 캐시에서도 못 찾으면, router dns 서버에 직접 조회하게 된다. 

dns 서버에서 해당 도메인 네임에 해당하는 ip주소(컴퓨터들이 인터넷 상에서 서로를 인식하기 위해 지정받은 식별용 번호)를 찾아 사용자가 입력한 url정보와 함께 전달한다. 페이지 url정보와 전달받은 ip주소는 http프로토콜을 사용하여 http 요청 메시지를 생성하고, 이 요청 메시지는 tcp 프로토콜을 사용하여 인터넷을 거쳐 해당 ip주소의 컴퓨터로 전송된다. 이렇게 도착한 http요청 메시지는 http프로토콜을 사용해 웹페이지 url정보로 변환되어 url정보에 해당하는 데이터를 검색한다. 검색된 웹페이지 데이터는 또다시 http프로토콜을 사용하여 http응답 메시지를 생성하고 tcp프로토콜을 사용하여 인터넷을 거쳐 컴퓨터로 전송된다. 

브라우저가 응답을 받으면, 브라우저가 가지고 있는 파서를 이용해 html문서를 브라우저가 이해할 수 있는 dom 트리 형식으로 파싱한다. 그 다음 css를 파싱하여 스타일 구조체의 형식 즉, cssom으로 만든 뒤, dom과 cssom을 실제 화면에 표현하기 위한 데이터 구조인 렌더링 트리로 변환한다. 해당 렌더링 트리를 그리고 화면에 표시한다. 

먼저, url을 결정된 규칙에 따라 분석하고 그 의미에 따라 http 리퀘스트 메시지를 만든다. 다음으로, 웹 서버의 ip 주소를 dns 서버에 조회한다. 넘겨받은 메시지에는 url이 있고 여기에 도메인 명이 존재한다.이 도메인 명을 기준으로 ip 주소를 조회한다. 알게 된 ip 주소를 os에 알려주면서 메시지 전달을 의뢰한다. ip주소와 http 리퀘스트 메시지를 프로토콜 스택에 전달하여 메시지 송신을 의뢰해야 하는데, 이 때 socket 라이브러리를 사용한다. 프로토콜 스택 내부는 데이터 송수신을 담당하는 tcp, 패킷 송수신 동작을 제어하는 ip로 나누어져 있다. 프로토콜 스택은 내부에 메모리 영역을 갖고 있어서 그곳에 통신 동작을 제어하기 위한 제어정보를 기록한다. 이 메모리 영역을 소켓이라고 할 수 있다. 대표적인 제어정보로는 통신 상대의 ip주소, 포트번호, 통신 동작의 진행상태 등을 포함한다. 

클라이언트가 서버에게 접속을 요청하는 패킷을 보낸다. 서버는 클라이언트의 요청을 받고 요청을 수락한다는 패킷을 발송한다. 클라이언트는 서버의 수락 응답을 받고 패킷을 서버로 보내면 연결이 성립된다. 

- http 프로토콜이란 클라이언트와 서버가 주고받는 메시지의 내용이나 순서를 정한 일종의 약속. 클라이언트에서 서버에게 보내는 메시지 안에는 '무엇을', '어떻게'에 대한 내용이 담겨져 있다. '무엇을'에 해당하는 것이 URI이고, '어떻게'에 해당하는 것이 메소드이다. 추가적으로 보낼 정보가 있을 때는 헤더 파일을 이용한다. 리퀘스트 메시지가 웹 서버에 도착하면 웹 서버는 메시지 내용을 해독한다. '무엇을', '어떻게'에 대한 동작을 하고, 결과 데이터를 응답 메시지에 저장한다. 이 응답 메시지의 맨 앞부분에는 실행결과에 대한 status code가 들어있고, 그 뒤로 헤더파일과 데이터가 이어진다. 이 응답 메시지가 클라이언트에게 전달되며 브라우저는 응답 메시지를 받아 화면에 표시하면서 http 동작이 끝난다.
- UDP: UDP를 사용한 것에는 DNS가 있다. 사전에 설정이 필요하지 않으며 그 후에 해제가 필요하지 않다. 짧은 요청을 보내고 짧은 응답을 기대한다.

13. 타입스크립트 도입 시 장점

type이 지원되면 정확함을 보장할 수 있고, 리팩토링이 쉽다. 자바스크립트의 var같은 자료형 대신, string, number 같은 자료형을 지정함으로써 안정성을 확보한다. 컴파일 과정에서 타입 검사를 통해 에러없이 안정성이 확보되면 타입을 제거하고 최종적으로 js코드를 생성한다. 또한 타입이 생김으로써 타입에 대한 예외 처리를 하지 않아도 된다. 그리고 js엔진의 최적화를 돕는다. js엔진은 자주 호출되는 함수(hot function)에 대해 최적화를 수행하는데, 예를 들면, JS엔진에서 인자 a,b가 정수형으로 들어와 더해지는 함수가 자주 호출되는 것을 확인했다면 앞으로도 a,b는 정수가 들어온다는 것을 가정하고 최적화를 수행한다. 그러다 갑자기 string이 들어오면 최적화가 해체되게 된다. js엔진에서 최적화와 해체 작업이 반복되면 성능이 저하되는데, ts에서는 타입을 지정하고 컴파일 과정에서 검사하기 때문에 최적화 해체가 발생할 상황이 적어진다. 

14. 가상돔과 돔과의 차이, 도입배경

돔이란, html내에 들어있는 요소를 구조화 객체 모델로 표현하는 양식으로, 트리의 부모 자식 관계로 형성된다. 일반적으로, 돔에는 수천개의 노드가 존재할 수 있고, 업데이트를 위해 복잡한 처리과정이 필요하게 되는데, 이로 인해 브라우저의 속도가 느려지게 된다. 현재까지 웹에서는 돔 변경에 맞춰 렌더링도 여러번 하게 되는 구조였는데, 이 비용이 너무 커서 도입된 것이 가상 돔이다. 이는 돔의 복사본을 메모리 내에 저장하여 사용하고, 변경사항을 가상의 위치에서 처리하고 실제 돔의 조작을 최소화한다. 따라서 직접 돔을 핸들링하지 않아 퍼포먼스를 향상시킨다. 이러한 방법으로 한 번의 상태 변경으로 여러 곳의 돔이 변경된다고 해도 가상화된 돔에 전부 변경하고 실제 돔과 비교해서 변경된 부분을 한번에 적용시키는 방식으로 돔의 렌더링 횟수를 줄이게 된다. 

15. react lifecycle API 설명

lifecycleAPI는 컴포넌트가 돔 위에 생성되기 전 후 및 데이터가 변경되어 상태를 업데이트하기 전 후로 실행되는 메소드이다. 컴포넌트를 생성할 때는 constructor → componentWillmount(컴포넌트가 돔 위에 만들어지기 전에 실행) → render → componentDidMount(컴포넌트가 만들어지고 첫 렌더링을 다 마친 후 실행) 순으로 진행된다. 컴포넌트를 제거할 때는 componentWillUnmount만 실행된다. 

컴포넌트의 prop이 변경될때는 componentWillReceiveProps(컴포넌트가 prop을 새로 받았을 때 실행된다. prop에 따라 state를 업데이트 해야 할 때 사용하면 유용하다) → shouldComponentUpdate(prop혹은 state가 변경되었을 때, 리렌더링을 할지 말지 정하는 메소드) → componentWillUpdate → render → ComponentDidUpdate 순으로 진행된다. state가 변경될 때는 props와 비슷하지만 shouldComponentUpdate부터 시작된다. 

16. props와 state

props는 부모 컴포넌트가 자식 컴포넌트에게 주는 값이다. 자식 컴포넌트에서는 props를 받아오기만 하고, 받아온 props는 직접 수정 할 수는 없다. state는 컴포넌트 내부에서 선언하며 내부에서 값을 변경할 수 있다.
