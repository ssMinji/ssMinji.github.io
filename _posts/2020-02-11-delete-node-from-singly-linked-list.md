### 노드를 가리키는 변수만 주어졌을 때 링크드리스트에서 노드 삭제하기
------

예를 들면, 입력값은 아래에서의 변수 ```b```처럼 될 수 있다. 

```python
class LinkedListNode(object):
    def __init__(self, value):
        self.value = value
        self.next = None

a = LinkedListNode('A')
b = LinkedListNode('B')
c = LinkedListNode('C')

a.next = b
b.next = c

delete_node(b)
```



#### Gotchas
------

우리는 이것을 *O(1)* 시간과 공간으로 할 수 있을 것이다. 하지만 우리가 원하는 정답은 조금 까다롭기 때문에, 부작용이 있을 수 있을 것이다. 



#### Breakdown
------

아마도 우리가 지우고자하는 노드를 마주칠때까지 리스트의 처음부터 순회하는 방법을 사용하고 싶을 수도 있다. 하지만 이 문제에서, 우리는 어디가 리스트의 처음인지 알 수 없다. (우리는 우리가 삭제하고자하는 노드에 대한 참조값만을 가지고 있다.)

음.. 잠시 생각해보자. 첫 번째 노드에 대한 참조값을 알고있는 일반적인 상황에서 링크드리스트에서 노드를 삭제하려면 어떻게 해야하는가?

우리가 지우고자 하는 노드를 스킵하기 위해 *이전* 노드의 포인트를 바꿔버리면, 이전 노드의 포인트는 곧바로 삭제될 노드의 *다음* 노드를 가리키게 될 것이다.  즉 만약 다음이 **노드를 삭제하기 전** 노드 상태라면: 

![deleteNodeBefore](https://user-images.githubusercontent.com/44194800/74215839-d734b980-4ce5-11ea-86fe-020e56baac66.PNG)

아래의 그림이 **삭제 후** 노드 상태가 될 것이다: 

![deleteNodeAfter](https://user-images.githubusercontent.com/44194800/74215888-03503a80-4ce6-11ea-9151-4a712d31c65c.PNG)

따라서 우리는 현재 노드를 스킵하고 바로 다음의 노드를 가리키는 방법을 알아야 한다. 하지만 우리는 이전 노드에조차 접근할 수 없다! 

이전 노드의 포인터를 변경하는 것 말고, **이전 포인터의 value에서 다음 포인터의 value로 건너뛸 수 있는 다른 방법이 있을까?**

만약 현재 노드를 지우는 대신에 **수정**한다면 어떨까?



#### Solution
------

우리는 인풋 노드의 다음 노드에서 ```value```와 ```next```값을 받아 인풋 노드에 복사한다. 이제 인풋 노드의 이전 노드는 효율적으로 인풋 노드의 예전 value값을 스킵하게 된다! 

따라서 예를 들면, 아래의 그림이 **함수를 호출하기 전** 링크드리스트의 모습이라면:

![beforeFunctionCall](https://user-images.githubusercontent.com/44194800/74218807-4f9f7880-4cee-11ea-910e-c4ac09ba201e.PNG)

아래의 그림이 **함수를 호출한 후**의 링크드리스트의 모습이다:

![afterFunctionCall](https://user-images.githubusercontent.com/44194800/74218908-968d6e00-4cee-11ea-9fd7-416c932ebcd3.PNG)

C같은 언어에서는 우리가 복사해온 노드는 더 이상 사용하지 않을 것이기 때문에 해당 노드를 직접 삭제해주어야 한다. 여기서는 우리는 파이썬의 가비지 콜렉터(garbage collector)가 그 일을 하게끔 할 것이다. 

```python
def delete_node(node_to_delete):
    # 우리가 스킵하고자 하는 인풋 노드의 다음 노드를 가져온다.
    next_node = node_to_delete.next
    
    if next_node:
        # 인풋 노드의 value와 pointer를 
        # 다음 노드의 value와 pointer값으로 바꾼다.
        # 이전 노드는 이제 인풋 노드를 스킵할 수 있다. 
        node_to_delete.value = next_node.value
        node_to_delete.next = next_node.next
    else:
        # 마지막 노드를 삭제하려 하는 경우
        raise Exception("마지막 노드는 이 방법으로 삭제 불가!")
```

**하지만 주의해야 할 점은 - 이 방법에는 몇몇 잠재적인 문제들이 존재한다는 것이다:**

**먼저, 이 방법으로는 리스트의 *마지막* 노드를 삭제할 수 없다.** 우리는 삭제하고자 하는 노드의 value값을 ```None```으로 바꿀 수 있지만, 두번째에서 마지막까지의 노드의 next 포인터는 그것이 ```None```이여야함에도 불구하고 여전히 그 노드를 가리키고 있을 것이다. 우리가 이 마지막, ```None```  value값을 가진 "삭제된" 노드를 "dead 노드" 혹은 "sentinel 노드"로 처리하고 노드를 탐색하는 코드가 이런 노드를 만났을 때 멈추게하도록 코드를 수정한다면, -  이 방법이 먹힐 수도 있다. 여기서의 트레이드오프는 우리는 value값으로 ```None```을 가진 non-dead 노드를 가질 수 없다는 것이다. 대신에, 이러한 경우에 대해 예외처리를 해주어야 한다.

두번째로, 이 방법은 몇몇 예상치못한 부작용을 일으킬 수 있다. 예를 들면, 우리가 다음과 같이 실행한다고 해보자:

```python
a = LinkedListNode(3)
b = LinkedListNode(8)
c = LinkedListNode(2)

a.next = b
b.next = c

delete_node(b)
```

여기에는 두가지 잠재적인 부작용이 존재한다:

1. **인풋 노드에 대한 모든 참조가 인풋 노드의 다음 노드로 재설정되어버린다.** 위의 예에서, 우리는 변수 b에 할당된 노드를 "삭제"했는데, 실제로는 우리가 단지 그 노드에게 새로운 2라는 value와 새로운 next를 준 것 뿐이다! 만약 코드 어딘가에 b를 가리키는 또다른 포인터가 존재하고 그것이 예전 value인 8을 갖는다고 가정해버리면, 그것은 버그를 발생시킬 것이다. 
2. **인풋 노드의 *기존 다음* 노드에 대한 포인터가 존재한다면, 그 포인터들은 이제 "댕글링" 노드를 가리키게 된다.** (댕글링 노드란, 리스트를 따라가도 더 이상 도달할 수 없는 노드를 이야기한다.) 위의 예에서, 노드 c가 이제 댕글링 노드이다. 만약 우리가 c를 변경한다면, 우리는 리스트의 처음부터 끝까지 따라간다해도 절대 그 새로운 값과 마주칠 수 없다. 



#### Complexity
------

*O(1)* 시간, *O(1)* 공간



#### Conclusion
------

이 문제는 해답이 완벽하지 않다. 왜냐하면 리스트를 "In-place(인플레이스)"에서 수정하기 때문에 주변 시스템의 다른 부분을 망가뜨릴 수가 있다. 이것을 "부작용(side effect)"이라 한다. 

In-place 알고리즘이란 원본 데이터 공간만을 사용하는 것을 말하는데, 이것이 시간 그리고(혹은) 공간을 아껴줄 수 있지만, 이 방법에는 리스크가 있다. 만약 인터뷰에서 in-place modification을 사용하게 된다면, 면접관에게 실제 시스템에서는 나머지 코드 부분에서의 부작용에 대해 신중히 고려할 것임을 말해주자~.~
